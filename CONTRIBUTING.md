# Contributing

Cardano Toolcheck is driven entirely by JSON. To add or change a tool, edit the
data — not the components. To make a tool's features show as *verified* in the
matrix, add tests for it (see "Add feature tests" below).

## Add a tool

1. Fork and clone the repo, then `npm install`.
2. Add an entry to [`data/tools.json`](./data/tools.json). Copy an existing
   entry as a template and fill in every field (see the table below). The
   `category` must match an `id` in `categories.json` and every feature in
   `features` must match an `id` in `features.json`.
3. If the tool is an off-chain SDK, add its rows to the matrices in
   [`data/compatibility.json`](./data/compatibility.json).
4. (Optional) Drop a logo in `public/logos/` and set the `logo` path.
5. Run `npm run build` to confirm the static export succeeds.
6. Open a PR.

### Required fields

| Field | Notes |
|-------|-------|
| `id` | unique kebab-case slug |
| `name`, `tagline`, `website`, `maintainer` | display basics |
| `repo`, `logo` | nullable |
| `category` | must exist in `categories.json` |
| `languages`, `maturity` | `maturity` ∈ production / active / experimental / legacy / deprecated |
| `features` | array of ids from `features.json` |
| `llmDescription` | 200–500 words shown on the tool-detail page. Explain what it does, what it does **not** do, how it relates to other tools, and any Cardano-specific context. |
| `whenToUse`, `skipUntil` | beginner-facing guidance |
| `dependsOn`, `optionalDeps`, `usedBy` | arrays of tool ids |
| `providers`, `smartContractCompat` | for SDKs only, else `null` |
| `isRecommendedDefault`, `recommendedFor`, `notRecommendedFor` | editorial recommendation |
| `healthCheck` | `{ type, repo, additionalChecks }` (see the health check types in the README/spec) |

## Add feature tests

A matrix cell only shows as *verified* when a real test exercises that feature and
passes. To add (or improve) a tool's verification:

1. Create `tests/<toolId>/` (the directory name must equal the tool's `id`) as a
   normal project in the tool's language — an npm/`vitest`, `pytest`, `cargo`, or
   Aiken project. See `tests/mesh`, `tests/pycardano`, `tests/aiken` for templates.
2. Add a `tool.test.json` descriptor: set `ecosystem`, the `runner` command, and a
   `tests` array where each entry lists the `features` id(s) it verifies, a `depth`
   (`offline` = no network; `devnet` = needs the local Yaci devnet), and a `match`
   string identifying the native test case.
3. Only reference features the tool actually claims in `tools.json`.
4. Run it locally: `npx tsx scripts/test-check.ts --tools <toolId>` then
   `npx tsx scripts/combine-test-results.ts`. CI re-runs only the tools you changed.

Keep tests from the **consumer's** perspective — install the published package and
call its public API, the way a dApp developer would.

## Don't hand-edit generated files

`data/health-results.json`, `data/test-results.json`, and
`data/test-results/<toolId>.json` are generated by CI scripts and overwritten —
don't edit them by hand.
